这篇文章整理的是 ZRAM、LMKD、kswapd 与后台留存之间的关系。原始灵感来自酷安 @Kisaratan 的[关于 ZRAM 压缩算法与后台留存](https://www.coolapk.com/feed/70915074?s=ZjRjMDc0ODAxYmMyZTBlZzZhM2U4ZmMyegi1631)，这里按技术脉络重新梳理，并补充 Android 与 Linux 官方文档里的相关机制。

后台留存不是单纯“内存越大越好”，而是一个动态平衡：系统要在应用体验、内存回收速度、功耗、卡顿和杀进程之间做取舍。ZRAM 能提高可用内存弹性，LMKD 决定什么时候杀后台，kswapd 决定什么时候提前回收，MGLRU 则影响内核找到可回收页面的效率。

## swap 是什么

swap 是 Linux 的内存交换机制。它的作用是在物理 RAM 紧张时，把一部分暂时不活跃的匿名页换出到后端设备里，等这些页面再次被访问时再换入内存。

这里的“匿名页”通常指应用运行时动态创建、没有普通文件直接对应的内存，比如堆对象、运行状态、部分 UI 数据等。

普通 swap 后端一般是：

- swap 分区
- swapfile 文件
- 压缩内存块设备，例如 ZRAM

换出和换入可以这样理解：

```text
RAM 中的冷匿名页
  ↓
swap out
  ↓
写入 swap 后端
  ↓
再次访问时 page fault
  ↓
swap in 回 RAM
```

swap 的意义不是凭空变出内存，而是把“暂时不活跃的数据”移出普通内存，让前台或新启动任务有空间运行。代价是换入换出会带来延迟：如果后端是闪存，会有 I/O 延迟；如果后端是 ZRAM，会有压缩和解压 CPU 开销。

## ZRAM 是什么

ZRAM 是 Linux 内核里的压缩块设备。它常被 Android 当作 swap 后端使用。

普通 swap 会把内存页换出到闪存或硬盘，而 ZRAM 不把数据写到存储设备上，而是把内存页压缩后继续放在 RAM 里。

可以这样理解：

```text
普通内存页
  ↓
被选为冷页
  ↓
压缩
  ↓
存入 ZRAM
  ↓
需要时再解压回普通内存
```

它的价值在于：如果 1GB 冷数据压缩后只占 300MB RAM，那么系统就相当于多回收出了约 700MB 空间。

但 ZRAM 不是免费午餐。压缩和解压都需要 CPU，压缩后的数据也仍然占用 RAM。如果压缩算法太慢，内存回收跟不上分配压力，系统仍然可能卡顿或杀后台。

## ZRAM 和普通 swap 的区别

ZRAM 和普通 swap 走的是类似的页面交换逻辑，但后端不同。

| 项目 | 普通 swap 分区 / swapfile | ZRAM |
| --- | --- | --- |
| 数据位置 | 闪存或磁盘 | RAM 内的压缩池 |
| 主要成本 | I/O 延迟和闪存写入 | CPU 压缩/解压 |
| 速度 | 受存储性能影响大 | 通常更快，取决于算法和 CPU |
| 寿命影响 | 可能增加闪存写入 | 不直接写闪存 |
| 容量来源 | 存储空间 | 物理 RAM 的一部分 |

Android 手机上更常见的是 ZRAM，因为手机闪存延迟和寿命都不适合承担大量传统 swap 写入，而 ZRAM 可以用 CPU 开销换取更快的内存回收与更少的闪存写入。

不过 ZRAM 仍然占用 RAM。它适合压缩冷数据，不适合把所有内存压力都硬塞进去。如果 ZRAM 被填满，回收速度跟不上，LMKD 和 OOM 路径仍然可能介入。

## ZRAM 大小不等于实际占用

很多人会把 “ZRAM 开了 8GB” 理解成 “直接占用 8GB 内存”，这是误解。

ZRAM 设备大小表示它最多能容纳多少未压缩前的逻辑数据。实际占用取决于：

- 有多少页面真的被换入 ZRAM。
- 这些页面能压缩到什么比例。
- ZRAM 元数据和内存池管理开销。
- 当前使用的压缩算法。

例如，逻辑上换入 1000MB 匿名页，如果压缩后占 280MB，那么实际 RAM 占用主要接近 280MB 加上额外管理开销，而不是直接占 1000MB。

所以，ZRAM 大小影响的是“最多能换出多少数据”，压缩率影响的是“换出后实际占多少 RAM”。

把 ZRAM 上限调高，并不代表这部分空间会立刻被占用，也不代表 CPU 会因为“空着的 ZRAM”而空转。没有页面被换入 ZRAM 时，就没有对应的压缩和解压工作；未使用的额度只是一个可用上限。

真正产生开销的是页面被换出和换入的过程：

- 换出时需要压缩，消耗 CPU。
- 换入时需要解压，也消耗 CPU。
- 压缩后的页面会占用一部分 RAM。
- ZRAM 元数据和内存池管理有少量额外开销。

因此，合理调高 ZRAM 更像是给系统留出“内存压力来临时的缓冲空间”。只要没有实际换入那么多数据，未使用部分不会凭空增加持续 CPU 开销。

## 压缩算法的取舍

ZRAM 常见算法包括 `lz4`、`lzo`、`lzo-rle`、`zstd` 等。它们的取舍大致如下：

| 算法 | 特点 | 适合方向 |
| --- | --- | --- |
| `lz4` | 压缩和解压速度快，压缩率中等 | 追求回收速度和响应 |
| `lzo` | 速度较快，表现均衡 | 保守稳定 |
| `lzo-rle` | 对重复数据友好，速度也不错 | Android 常见默认方案 |
| `zstd` | 压缩率高，但 CPU 开销更大 | 追求更高压缩率，需看实现和等级 |

后台留存不是只看压缩率，也不是只看吞吐量，而是要看“有效内存回收速度”。

一个简化估算：

```text
有效回收速度 = 压缩速度 × (1 - 压缩后占用比例)
```

假设某算法压缩速度是 400MB/s，压缩后占用 25%，那么有效回收速度约为：

```text
400 × (1 - 0.25) = 300MB/s
```

这比单看“压缩速度 400MB/s”更接近后台留存场景，因为系统真正需要的是尽快腾出可用内存。

## 为什么基准测试不等于实际 ZRAM 性能

压缩库基准测试通常只测算法本身，比如一段数据压缩和解压有多快。

但 ZRAM 的完整路径更复杂：

```text
内存页选择
  ↓
swap 路径
  ↓
ZRAM 元数据管理
  ↓
zsmalloc 内存分配
  ↓
压缩算法
  ↓
写入压缩池
```

当算法本身很慢时，比如高等级 `zstd`，瓶颈主要在压缩算法。

当算法本身很快时，比如 `lz4`、`lzo-rle`，瓶颈可能就不再是算法，而是 ZRAM 驱动、内存池管理、页交换路径和内核内存子系统。因此，换一个更快版本的算法，不一定能继续明显提升 ZRAM 实测速度。

## kswapd 在做什么

Android 低内存时，内核会通过页面回收来腾出内存。这里最重要的后台线程之一就是 `kswapd`。

它大致负责：

1. 发现空闲内存低于某个水位线。
2. 扫描内存页，找出较冷的页面。
3. 文件页可以直接回收，因为它们能从存储重新读取。
4. 匿名页没有文件后备，需要写入 swap，ZRAM 就是常见后端。

匿名页包括应用运行时动态创建的数据，比如堆对象、UI 状态、运行时内存等。它们被换入 ZRAM 后，系统就能释放原本占用的普通内存。

Android 手机通常是单 NUMA 设备，常见情况下只有一个主要 `kswapd` 线程在承担页面回收压力。即使 ZRAM 支持多个压缩流，实际回收路径也可能因为 `kswapd` 的串行处理而无法充分吃满多线程能力。

这也是为什么单线程压缩和解压速度仍然很重要。

## kcompressd 的意义

一些第三方内核会引入类似 `kcompressd` 的思路，把压缩请求从 `kswapd` 中拆出去异步处理。

这样做的目标是：

- 减少 `kswapd` 被压缩操作阻塞的时间。
- 让 `kswapd` 更快扫描和提交可回收页面。
- 提升内存压力下的回收效率。

它的本质不是让 ZRAM 神奇变大，而是改善回收链路的并发和响应速度。是否稳定、是否适合日常使用，要看具体内核实现。

## 后台为什么会被杀

Android 后台留存通常会被三层机制影响：

1. 厂商自己的后台管理策略。
2. Android 用户态的 LMKD。
3. Linux 内核 OOM Killer。

厂商策略通常和电量、温控、后台限制、应用白名单有关。不同系统差异很大。

LMKD 是 Android 的低内存杀进程守护进程。Android 11 之后，它主要结合 PSI 等内存压力信息来判断系统是否出现内存分配停滞。它不只是看“还剩多少空闲内存”，更关心“内存分配能不能及时完成”。

OOM Killer 是更底层的兜底机制。当内核已经无法满足内存分配，且回收也失败时，才会强制杀进程。

## memcg 在哪里

memcg 是 memory cgroup，也就是内存控制组。它不是某个压缩算法，也不是单独的杀后台服务，而是 Linux cgroup 里负责“按组统计、限制和回收内存”的机制。

在 Android 里，应用进程通常会被放进不同的 cgroup。这样系统就能按进程组统计内存使用、施加限制，并把前台、后台、缓存进程放在不同的资源管理语境里。

memcg 主要影响这些事：

- 按应用或进程组统计内存占用。
- 对某一组进程施加内存上限或保护策略。
- 在组内触发回收，而不是只看全系统内存。
- 为 LMKD、系统服务和调度策略提供更细粒度的内存状态。

可以把关系理解成：

```text
swap：冷匿名页临时换出的通用机制
memcg：谁用了多少内存、属于哪个组
PSI：系统或分组是否出现内存压力停滞
kswapd / reclaim：尝试回收页面
ZRAM：匿名页换出后的压缩后端
LMKD：压力持续时选择杀掉哪些进程
```

所以，memcg 更像“分组账本和控制边界”。它让系统不只是看到“全局内存紧张”，还可以看到“某个应用组或后台组正在消耗多少内存”。这对后台管理很重要，但最终是否杀进程，通常还是由 LMKD、厂商策略或内核 OOM 路径来执行。

## PSI 和 LMKD 的关系

PSI 是 Pressure Stall Information，也就是压力停滞信息。它能反映 CPU、内存、I/O 资源压力导致任务停滞的时间比例。

对后台留存来说，内存 PSI 很关键。它能告诉 LMKD：系统是不是已经因为内存压力而让任务等待太久。

LMKD 的思路可以简化成：

```text
内存分配变慢或停滞
  ↓
PSI 记录压力时间
  ↓
超过阈值
  ↓
LMKD 按策略选择进程
  ↓
杀掉相对不重要的进程释放内存
```

所以提升后台留存有两个方向：

- 让 LMKD 更能容忍短时间内存压力。
- 提升内核回收速度，让分配请求尽快得到内存，避免 PSI 压力继续升高。

## LMKD 参数应谨慎调整

LMKD 支持一些系统属性调整行为，例如：

```text
ro.lmk.psi_partial_stall_ms
ro.lmk.psi_complete_stall_ms
ro.lmk.swap_util_max
ro.lmk.thrashing_limit
ro.lmk.thrashing_limit_critical
ro.lmk.thrashing_limit_decay
ro.lmk.kill_heaviest_task
ro.lmk.kill_timeout_ms
```

这些参数能影响 LMKD 对内存压力、swap 使用、页面缓存抖动和杀进程节奏的判断。

但不建议盲目把阈值拉得很高。LMKD 杀后台虽然烦，但它也是系统避免严重卡顿和内存耗尽的保护机制。如果几乎不让它工作，系统可能从“杀后台”变成“前台卡死”。

更合理的方向是：

- 适当提高压力容忍度。
- 延长连续杀进程间隔，给内核回收留时间。
- 保证 ZRAM 能被充分利用。
- 不要直接禁用或删除 LMKD。

尤其不要尝试删除 `/system/bin/lmkd` 或强行冻结 LMKD。它是系统核心服务，禁用可能导致系统异常甚至无法正常进入桌面。

## 内存水位线

Linux 内核有几条重要的内存水位线，常见理解是：

| 水位线 | 含义 |
| --- | --- |
| High | 回收到这里，后台回收可以停下 |
| Low | 低于这里，唤醒 `kswapd` 异步回收 |
| Min | 低于这里，可能触发直接回收 |

当空闲内存低于 Low，`kswapd` 会在后台开始回收，直到空闲内存回到 High。

如果回收跟不上，空闲内存继续跌到 Min 附近，就可能触发直接回收。直接回收会阻塞正在申请内存的任务，表现就是界面卡住、应用短暂无响应。

如果直接回收仍然失败，才可能触发 OOM Killer。

## VM 参数如何影响回收

几个常见参数：

```text
/proc/sys/vm/min_free_kbytes
/proc/sys/vm/watermark_scale_factor
/proc/sys/vm/extra_free_kbytes
```

它们影响内存水位线的位置。

`min_free_kbytes` 是内核保留的最低空闲内存基础值。一般不建议乱改，内核默认值通常比较稳。设太高会浪费内存，也可能更早触发回收；设太低可能让系统更接近 OOM 风险。

`watermark_scale_factor` 决定水位线间隔，会影响 `kswapd` 何时开始工作。值更高，通常意味着更早回收，但也可能带来更多后台回收和功耗。

`extra_free_kbytes` 可以额外拉高空闲内存目标，让 `kswapd` 更早开始回收，换来更长的回收时间窗口。

这些参数的核心取舍是：

```text
更早回收 = 更容易保留后台 / 更少直接回收卡顿
但也可能 = 空闲内存保留更多 / 实际可用内存变少 / 功耗更高
```

因此不要把水位线调得过激。比较稳的思路是小幅调整，再观察后台、卡顿、功耗和发热。

## MGLRU 是什么

MGLRU 是 Multi-Gen LRU，也就是多代 LRU。它是 Linux 内核中更现代的页面回收机制。

传统 LRU 会维护活跃和非活跃页面列表，但在复杂负载下，找出真正该回收的页面并不总是高效。MGLRU 会把页面按访问新旧分成多个 generation，更细地判断哪些页面更冷。

它对后台留存的意义在于：

- 更准确地找出冷页。
- 降低无效扫描开销。
- 减少把热页错误回收导致的抖动。
- 提升内存压力下的回收效率。

简单说，MGLRU 不是直接“增加内存”，而是让内核更聪明地决定哪些页面该留下，哪些页面该回收。

## 怎样选择 ZRAM 算法

没有一个算法适合所有设备。

比较实用的判断方式是：

1. 如果设备 CPU 单核不强，优先考虑 `lz4` 或 `lzo-rle` 这类快速算法。
2. 如果内核的 `zstd` 实现足够新，并且使用快速等级，可能兼顾不错压缩率和速度。
3. 如果厂商内核对某个算法做了特殊优化，不要只看开源默认表现。
4. 不要只看压缩率，也要看压缩速度和解压速度。
5. 真实后台体验比单次跑分更重要。

ZRAM 算法的最终目标不是跑分好看，而是在内存压力来临时更快释放空间，并在应用切回前台时更快解压。

## 想提升后台留存可以看哪里

比较安全的排查顺序：

1. 先确认厂商后台策略有没有限制目标应用。
2. 查看 ZRAM 是否启用，大小是否明显过小。
3. 观察当前 ZRAM 算法和压缩率。
4. 关注内存压力下是否频繁触发 LMKD。
5. 看是否出现直接回收导致的明显卡顿。
6. 评估内核是否支持 MGLRU 等更好的回收机制。
7. 小幅调整 VM 水位线参数，并长期观察。

如果目标是“后台尽量多留”，可以适当提高回收提前量和 LMKD 容忍度。

如果目标是“前台绝对流畅”，就不要过度追求后台留存。留太多后台意味着更多内存压力，也可能让前台应用更容易遇到直接回收。

## 一个相对稳的思路

后台留存优化可以按下面思路组合：

- ZRAM 不要太小，至少要能承接一定匿名页。
- ZRAM 上限可以适当留余量，未使用部分不会让 CPU 空转。
- 算法优先选有效回收速度高的，不只看压缩率。
- LMKD 阈值可以适当放宽，但不要禁用。
- `kswapd` 可以更早开始回收，但水位线不要拉太高。
- 有 MGLRU 的内核通常更有优势。
- 所有调整都要用真实使用场景验证。

## 不建议做什么

这些做法风险较高：

- 直接禁用 LMKD。
- 为了“省 CPU”直接关闭 ZRAM。
- 把 LMKD 阈值拉到几乎不杀后台。
- 盲目把 `min_free_kbytes` 调很大。
- 只为了后台数量牺牲前台流畅度。
- 只看跑分，不看实际应用切换和卡顿。
- 复制别人的整套参数，不考虑自己的内核、内存和系统策略。

后台留存优化不是越激进越好。太保守会杀后台，太激进会卡前台。真正好的配置是在两者之间找平衡。

尤其不建议把“关闭 ZRAM”当成优化。关闭后确实少了压缩/解压开销，但也少了一层匿名页缓冲空间。内存压力上来时，系统可能更早进入直接回收、LMKD 杀后台，甚至更接近 OOM。对多数 Android 设备来说，保留合适大小和合适算法的 ZRAM，比直接关闭更稳。

## 小结

swap 是匿名页换出和换入的通用机制；ZRAM 是 Android 上常见的压缩内存 swap 后端，影响匿名页被压缩后的空间利用和回收速度；memcg 负责按进程组统计和限制内存；`kswapd` 影响内存回收启动和扫描效率；LMKD 影响什么时候杀后台；MGLRU 影响内核找冷页的准确性。

想提高后台留存，不能只改一个参数。更合理的思路是：选合适的 ZRAM 算法，保证回收速度；适当提前 `kswapd` 工作窗口；让 LMKD 对短时间压力更宽容；同时避免把系统推到直接回收和前台卡顿的边缘。
