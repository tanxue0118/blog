后台进程清理不是为了把后台清空，而是为了把系统从“还能跑”拉回“跑得顺”。

Android 早就有自己的兜底链路，LMKD、OOM、memcg、PSI 这些机制都在场。问题在于，这些机制更像最后一道保险，而不是日常秩序本身。厂商策略一旦把后台留得太松，应用又带着一串互相拉起的常驻链路，系统就会慢慢变成一种“表面在线、实际负债”的状态。内存被占着，页面在抖，I/O 在排队，前台的热启动、切回和重建就开始变钝。

如果你有墓碑，后台进程多，也顶多是占用内存高，进程都被冻结住了，但是如果你没墓碑，后台进程不仅占用内存高，还会一直占用CPU。

所以真正值得做的，不是杀死所有在后台的进程，而是把一些没用的垃圾进程给杀死。

## 为什么需要后台清理

后台清理的价值，首先不是省内存，而是减少系统在压力上升时的连锁反应。

在 Linux 和 Android 里，swap 的意义从来不是“凭空多出内存”，而是把一部分暂时不活跃的页面挪到别处，换来更平滑的周转。这个过程中，`kswapd` 要工作，缺页处理要工作，CPU 和 I/O 都会被卷进去。手机现在普遍是 UFS，热启动时重新 fork、重建进程、重载代码和资源，本来就会和回收、swap、文件缓存恢复抢 I/O。后台如果堆得太厚，前台请求就容易被挤到后面，最后表现出来的不是“内存不够”，而是切换卡、启动慢、滑动顿。

所以，后台清理的目标不是“后台越少越好”，而是“后台里真正该留的留下来，不该留的尽早清理”。

## 哪些进程更适合清理

并不是所有进程都适合被清理，在我所制作的模块中，我运用了三个方式判断哪些进程会被清理，首先我借鉴的是安卓的oom_adj_score机制，当 score≥800 时杀，还有当cpuset处于background时就杀以及黑名单里面的也杀。

| 类型 | 典型特征 | 处理倾向 |
| --- | --- | --- |
| 前台链路 | `top-app`、`foreground`、正在交互、正在播放、正在下载 | 默认保护 |
| 可疑后台 | 普通后台服务、临时 worker、辅助子进程、长时间不活跃但仍挂着的包 | 条件清理 |
| 明显可杀 | 缓存进程、空进程、反复唤起的残留链路、已经失去意义的后台壳 | 优先清理 |

真正麻烦的地方在于，很多应用不是一个进程，而是一整条链。

有的包主进程和推送进程绑定得很紧，杀一个会把另一个反复拉起来。有的应用主进程看着还在，实际真正占资源的是 WebView、渲染、下载、同步、辅助服务这一串子进程。还有一些应用退到后台后，只要消息、音频、下载、同步没有真正结束，就不能按“空闲垃圾”处理。

所以，清理对象不能只看 PID、score、cpuset，要看角色、关联关系、唤醒代价和是否会反复唤醒。

比如小米系统自带音乐，杀掉主进程后，音乐链路有时还会继续在后台跑，重新进入软件之后则会刷新并停播。QQ 更典型，接收消息的 MSF 进程和主进程往往是绑定的，杀掉其中一个，另一个很容易被反复拉起。对于接入了各家 push 的软件，后台往往只保留一个主进程就够了，像微信这种不愿意接各家 push 的，通常就得同时保活主进程和 push 进程。

## 为什么我会用 cpuset

cpuset 原本是 CPU 和内存节点摆放的工具，不是杀进程名单。

它的本意是告诉内核，某个任务应该在哪些 CPU 上跑，哪些内存节点可以用。
但在 Android 里，这个边界本身就很有价值，因为很多 ROM 会把进程放进不同的组里：`top-app`、`foreground`、`background`、`system-background`、`restricted` 之类。名字和层次会因版本、厂商实现而变化，但大方向通常一致，越靠前越该保，越靠后越像后台候选。

这也是 cpuset 能被拿来当筛子的原因。它不负责决定谁会被杀死，只负责告诉系统，谁本来就站在低优先级那一边。先用它缩小范围，再去看 `oom_score_adj`、历史活跃度、白名单、黑名单和当前压力，效率和准确度都会高很多。

可以把它理解成：

```text
cpuset 负责给出候选范围
oom_score_adj 负责给出清理偏好
LMKD 负责在压力上来时兜底
```

这三层叠起来，才像一套完整的后台治理语言。

## Android 原生机制本来在做什么

### `oom_score_adj`

`oom_score_adj` 的范围是 `-1000` 到 `+1000`。数值越大，越容易在 OOM 里被挑中；`-1000` 接近完全豁免。

它不是简单的“好人分数”或“坏人分数”，更像是一个偏好权重。内核最终看的是 badness 计算结果，而 `oom_score_adj` 会把偏好直接加进去。对于 Android 来说，这一层很重要，因为它允许系统把“前台、可感知、后台、缓存”这些语义，映射成可执行的回收倾向。

常见的经验分段大致可以这样看：

| 分数段 | 大致含义 |
| --- | --- |
| `-1000` | Linux 内核进程和 Android 极其核心的服务 |
| `700-800` | 后台进程 |
| `900-1000` | 缓存进程或空进程 |

这不是铁律，不同 ROM 和版本会有偏差，但拿来判断优先级通常够用。

### LMKD

Android 10 之后，`lmkd` 逐渐转向 PSI 作为主要的内存压力检测方式。它不再只是盯着“还剩多少空闲内存”，而是看任务因为内存短缺被拖住了多久。这个变化很关键，因为真正伤体验的，从来不是某个绝对数字，而是压力已经传导到交互侧。

换句话说，LMKD 关心的是“系统是不是已经开始卡”，而不是“现在纸面上还剩几 MB”。

### PSI

PSI 记录的是任务因为资源短缺而停住的时间比例。对内存治理来说，这比单看 free page 更接近真实体感。页面在等、任务在等、交互在等，这些等待本身就是压力。

### memcg

`memcg` 是 memory cgroup，负责按组统计、限制和回收内存。它让系统能把“谁在花内存”这件事拆到组级别，而不是只看全局总量。对于后台治理来说，这一层很像账本。

### 这些机制的关系

```text
cpuset：谁在什么资源池里
oom_score_adj：谁更该被留下
memcg：谁占了多少内存
PSI：压力是不是已经传导成卡顿
LMKD：压力持续时，谁先被处理
```

这套东西放在一起看，就会发现后台清理不是在和内核对着干，而是在顺着内核已有的语义，把“该不该留”提前做掉。


## 这套清理思路到底想解决什么

核心问题不是“杀不杀”，而是“先收拾谁、收拾到什么程度、什么时候停手”。

如果一个后台进程只是偶尔活跃，处理时机可以更保守；如果它已经长期躺在后台、还会反复唤醒、还会带着一串子进程一起复活，那就不能再把它当作普通常驻内容看待。更合理的做法，是先把最冷、最旧、最不该继续占位的对象挑出来，再按力度分层处置。

对于杀死进程来说，一种更稳妥的执行方式，是先 `SIGSTOP`，再 `SIGKILL`，魔威说这样子能够防止进程不断唤醒。

所以，我模块的后台清理更像一个分层动作：

```text
识别候选
  -> cpuset / cgroup 初筛
  -> 白名单 / 黑名单 / 前台保护
  -> 运行历史和重复拉起判断
  -> 先 SIGSTOP，必要时再 SIGKILL
  -> 短期保护，避免反复鞭尸
```

## 结语
一般来说，只要不杀死主进程，就不算强制停止该软件，重新加载只需要重新加载一次，估计是清理了webview的原因，但是这个问题也不大，当你进去使用功能后，被杀死的子进程也会随之被唤醒，我觉得用一次重新加载来换取更加清净的后台是可取的。
